Лабораторная работа по ASP.NET MVC.

Создание сайта - баг-трекера.
В баг-трекере есть следующие роли: Разработчик, Менеджер проекта, Администратор.


Проекты.

В системе можно создавать Проекты. Проекты характеризуются названием, описанием, списком задач, заказчиком, Менеджером. При создании проекта можно вручную указать название проекта, его описание, название (имя) заказчика, выбрать из списка пользователей с ролью "Менеджер проекта" Менеджера проекта.

Должна быть страница, где приведен в виде таблицы список всех Проектов, созданных в системе с возможностью перейти к просмотру конкретного проекта.
Страница просмотра информации о проекте должна выводить его название, описание, имя заказчика, ниже таблицей должен быть выведен список задач в этом проекте с возможностью перехода к конкретной задаче.
Опционально на этой странице можно реализовать фильтр задач по названию и сортировку по названию, статусу, стоимости.

Просматривать как список проектов, так и конкретный проект, может кто угодно.


Задачи.

В каждом Проекте есть Задачи.
Задачи характеризуются следующими свойствами: номер, название, описание, "назначена на" (какого пользователя), статус, стоимость.

Просматривать задачи может кто угодно.

Задачи могут создавать пользователи с ролью Менеджер проекта (он может делать это только в своих проектах) и Администратор. Для этого должна быть предусмотрена специальная кнопка на странице просмотра списка задач проекта.

При создании задачи должна быть возможность указать ее название, описание, указать в выпадающем списке Разработчика, на которого она будет назначена, указать стоимость. При успешном создании новой задачи она автоматически получает номер (в качестве его можно использовать int-овое поле Id) и открывается страница просмотра только что созданной задачи.

На странице просмотра задачи отображается номер задачи, название, описание, на кого назначена, текущий статус, стоимость. Также есть кнопка Редактировать, по которой открывается страница редактирования. На ней можно менять параметры задачи (кроме номера), в том числе менять Разработчика, на которого она назначена. Удаление задачи делать не нужно, т.к. в реальных баг-трекерах задачи не пропадают просто так. Кроме того, на странице просмотра должна быть возможность перевести задачу в один из доступных статусов. Доступность статуса определяется схемой ниже:

Задача создается в статусе "Открыта", далее статус может меняться только по стрелкам:
Открыта -> Готова к ревью;
Открыта -> Закрыта;
Готова к ревью <-> Переоткрыта;
Готова к ревью -> Закрыта;
Переоткрыта <-> Закрыта.

Изменять статус может пользователь, на которого она назначена, а также пользователь с ролью Менеджер проекта (опционально - только Менеджер того проекта, к которому относится задача), или Администратор системы.


Пользователи.

Авторизация и аутентификация пользователей должна осуществляться с помощью Identity. Администратор может просмотреть список пользователей в системе в виде таблицы.

Регистрация пользователей:
Первый вариант:
1. Добавлять новых пользователей может Администратор. На экране добавления нового пользователя Администратор может указать пользователю логин, пароль, выбрать роль (Разработчик, Менеджер проекта).
2. Пользователи могут сами регистрироваться в системе, в таком случае они должны получать роль Разработчик.

Второй вариант:
Пользователи могут при регистрации выбирать себе роль (Разработчик, Менеджер проекта).

Администратор в системе должен быть по крайней мере один, и он должен создаваться по умолчанию.

В качестве БД используйте localdb, в качестве ORM используйте Entity Framework.
